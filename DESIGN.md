# Threads ポスト収集拡張機能 設計書

X（Twitter）の「ついすぽ -Tweet Export-」を参考にした、**Threads** 向け Chrome 拡張機能の設計図です。

---

## 1. 概要

### 1.1 目的
- Threads 上で**特定アカウントのポストのみ**を検索し、引用・リプライ・コメントを除いた**純粋なポスト**を収集する
- スクロールに応じてポストをカウント・収集し、**CSV でダウンロード**できるようにする

### 1.2 対象ページ
- **Threads 検索ページ**  
  - 例: `https://www.threads.net/search?q=...` または検索結果が表示される URL
- 検索クエリで「from:@username」のような形でアカウント指定ができる前提（Threads の検索仕様に依存）

### 1.3 主な機能（要件との対応）

| 要件 | 機能 |
|------|------|
| 純粋ポストのみ検索 | 「純粋ポストで検索」ボタン（引用・リプコメントを除外する検索／フィルタ） |
| スクロールで収集 | スクロールに連動したポストの検出・重複除き・カウント表示 |
| CSV ダウンロード | 収集したポストを CSV でダウンロードするボタン |

---

## 2. アーキテクチャ

### 2.1 拡張機能の構成

```
threads-export-extension/
├── manifest.json          # Manifest V3
├── background.js          # サービスワーカー（必要に応じて）
├── content/
│   ├── content.js         # メイン: DOM 解析・収集・UI 注入
│   └── content.css        # 拡張用 UI のスタイル
├── popup/
│   ├── popup.html
│   ├── popup.js
│   └── popup.css
├── icons/                 # 16, 48, 128px
└── DESIGN.md              # 本設計書
```

### 2.2 動作フロー

1. ユーザーが Threads の検索ページを開く
2. 開発者ツールのタブ、またはページ内に「ついすぽ Threads 版」用 UI が表示される
3. **「純粋ポストで検索」** をクリック → 検索パラメータの付与またはフィルタ適用
4. ユーザーがスクロール → **Content Script** が DOM を監視し、新規ポストを検出・収集・カウント更新
5. **「Download CSV」** クリック → 収集済みデータを CSV 化してダウンロード

## 3. 機能詳細設計

### 3.1 「純粋ポストで検索」ボタン

**役割**
- 引用スレッド（quote）・リプライ（reply）・コメントを**検索結果から除外**する

**実装方針（2パターン想定）**

| 方式 | 説明 |
|------|------|
| **A. 検索時フィルタ** | Threads の検索 URL に「ポストのみ」に相当するクエリやパラメータがあれば付与する（公式 API/URL 仕様の調査が必要） |
| **B. 表示後フィルタ** | 検索結果を表示したあと、DOM 上で「リプライ」「引用」等のノードを非表示にし、タイムライン上では「ポストのみ」が並んでいるように見せる |

**UI**
- 検索ページの上部または開発者ツールパネル内に「純粋ポストで検索」ボタンを配置
- クリックで「ポストのみモード」ON/OFF を切り替え、ON のときは B の場合は DOM フィルタを適用

**補足**
- Threads の検索が「アカウントのポストのみ」をサポートするかは要調査。サポートしない場合は「全検索結果からリプライ/引用を除外して表示」の B が主になる。

---

### 3.2 スクロール連動の収集・カウント

**役割**
- ユーザーがスクロールするたびに、現在表示されている（またはこれまでに出現した）**ポスト**を収集
- 引用・リプライ・コメントは収集対象外（純粋ポストのみ）
- 重複は除外して件数をカウントし、UI に表示

**処理フロー**

1. **対象ページの判定**  
   - URL が Threads の検索/タイムライン系であることを確認してから処理を有効化

2. **ポストの検出**
   - Threads のタイムライン DOM 構造に依存
   - 一般的な想定:
     - 各ポストが何らかのコンテナ（例: `article`, `div[data-*]` など）で囲まれている
     - リプライ・引用は別の属性やクラス・ラベルで区別されている
   - **セレクタの設計**: 実際の Threads の HTML を調査し、「ポスト」「リプライ」「引用」を判別するセレクタを定義する（後述「Threads DOM 調査」参照）

3. **スクロール／表示の監視**
   - `IntersectionObserver` またはスクロールイベントで、ビューポートに新規に入ってきた要素を検出
   - 新規に出現した「ポスト」ノードだけを配列に追加（既存 ID などで重複排除）

4. **データの抽出（1ポストあたり）**
   - テキスト本文
   - 投稿日時
   - 投稿者（@username, 表示名）
   - いいね数・返信数・引用数（DOM に表示されていれば）
   - ポスト URL
   - その他、DOM から取得できるメタデータ

5. **カウント表示**
   - 収集件数を「収集: N 件」のように常に表示（開発者ツールパネル内またはページ内のフロート UI）

**注意**
- Threads は SPA のため、スクロールで動的に DOM が追加される。無限スクロールのたびに新ノードを監視対象に追加する必要がある。

---

### 3.3 CSV ダウンロード

**役割**
- これまでに収集したポスト一覧を CSV でダウンロードする

**CSV 項目案**

| 列名（例） | 説明 |
|------------|------|
| id | 拡張側で付与する一意 ID（重複排除用） |
| post_text | ポスト本文 |
| post_url | ポストの URL |
| author_username | 投稿者 @username |
| author_display_name | 投稿者表示名 |
| posted_at | 投稿日時（ISO 8601 推奨） |
| likes_count | いいね数 |
| replies_count | 返信数 |
| quotes_count | 引用数（取得可能なら） |
| collected_at | 拡張が収集した日時 |

**実装**
- Content Script 内で収集済み配列をループし、CSV 文字列を生成（改行・カンマ・ダブルクォートのエスケープを実施）
- `Blob` + `URL.createObjectURL` + `<a download>` でダウンロード、または `chrome.downloads` API を利用

**ファイル名例**
- `threads_posts_YYYYMMDD_HHmmss.csv`

## 4. UI 設計

### 4.1 配置パターン（ついすぽと同様）

- **開発者ツール内タブ**
  - Chrome 拡張の **devtools** 機能を使い、開発者ツールのタブに「Threads Export」などを追加
  - そのタブ内に「純粋ポストで検索」「収集: N 件」「Download CSV」を配置
- **ページ内フロート**
  - Content Script で検索ページに小さなパネルを注入し、同じボタン・カウント・Download を表示

どちらか、または両方対応可能。

### 4.2 ボタン・表示

- **純粋ポストで検索**: トグルまたは一度押すと「ポストのみモード」ON
- **収集件数**: 「収集: 123 件」のように数字をリアルタイム更新
- **Download CSV**: 青い「Download」ボタン（ついすぽと同様）

## 5. 技術メモ

### 5.1 Manifest V3

- `manifest.json` は Manifest V3 で作成
- `host_permissions` に `https://www.threads.net/*` を指定
- Content Script の `matches`: `["https://www.threads.net/*"]`
- 必要に応じて `scripting.executeScript` で動的注入

### 5.2 Threads DOM 調査（実装前に必須）

- 実際の Threads 検索ページの HTML を取得し、以下を特定する:
  - 1ポストを表すルート要素のセレクタ
  - リプライ・引用・「ポスト」を区別する属性・クラス・aria ラベル
  - 本文・ユーザー名・日時・いいね数などのサブセレクタ
- 調査結果に基づき、`content.js` 内のセレクタ定数を更新する設計にする。

### 5.3 利用規約・倫理

- 取得するのは**画面に表示されている公開情報のみ**
- 過度なリクエストやスクレイピングは避け、ユーザーの手動スクロールに合わせた収集に留める
- 利用規約の変更に注意し、必要に応じて機能説明を更新する

---

## 6. 実装順序の提案

1. **環境準備**: `manifest.json`、Content Script のスケルトン、Threads 検索ページでのみ動作させる
2. **DOM 調査**: Threads の検索結果ページでポスト/リプライ/引用のセレクタを確定
3. **収集ロジック**: スクロール監視 + ポスト検出 + 重複除き + データ抽出
4. **カウント表示**: 収集件数の表示
5. **純粋ポストで検索**: 検索パラメータまたは DOM フィルタの実装
6. **CSV 出力**: 収集データの CSV 生成とダウンロード
7. **UI 整備**: 開発者ツールタブ or ページ内パネルの見た目・操作性の調整

| 項目 | 内容 |
|------|------|
| 名前案 | ついすぽ Threads 版 / Threads ポスト Export など |
